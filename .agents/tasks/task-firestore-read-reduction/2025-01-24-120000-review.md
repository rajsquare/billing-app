# Firestore read reduction for billing app

This change reduces Firestore reads in a client-side billing app by date-bounding the bills and daybook snapshot listeners, caching the product catalog in localStorage, debouncing live draft syncs with content-hash deduplication, adding a visibility guard to daybook rendering, and auto-cleaning stale live drafts. The approach is pragmatic: no architectural changes, all optimizations layered into the existing single-file structure.

**Watch for:** The `getStartOfTodayTimestamp()` helper is called at module top-level (lines 62, 69) but depends on `_todayFmt`, a `const` declared at line 116 — this is a temporal dead zone crash that prevents the app from loading (confirmed). The date-bounded queries are also evaluated once at page load and become stale if the session crosses midnight (likely operational issue for overnight use).

## High-level view

The query constraints (`where("createdAt", ">=", ...)` and `limit(200)`) cut the snapshot listener scope from "all bills ever" to "today's bills, capped at 200". This is the highest-impact optimization for read volume. However, the timestamp is computed once at module load, creating both a TDZ crash and a staleness problem for long-lived sessions.

The product catalog cache uses a date-keyed localStorage entry to skip the Firestore read entirely when the cached date matches today (IST). The gap is that `localStorage.setItem` can throw `QuotaExceededError` if the catalog is large, and that error propagates to the outer catch, breaking the entire load path.

The debounced sync replaces three direct `syncLiveDraft()` call sites with a 1-second debounce plus a djb2-style content hash. Draft deletion is now delayed by 1 second (peers see a "ghost" draft briefly after a bill is cleared), but all edit paths are correctly covered.

The stale draft cleanup runs inside the live drafts `onSnapshot` callback, throttled to once per 30 seconds, deleting other sessions' drafts older than 2 minutes. A session experiencing a brief network outage (>2 min) could have its draft deleted by a peer before it reconnects.

## Temporal dead zone crash at module initialization

`getStartOfTodayTimestamp()` is a `function` declaration (hoisted), but its body references `_todayFmt`, which is a `const` declared at line 116. The calls at lines 62 and 69 execute during module evaluation, before line 116 is reached. In ES modules, `const` bindings exist from the start of the scope but are in the TDZ until their initializer runs. Accessing `_todayFmt` at line 332 (`_todayFmt.format(now)`) throws `ReferenceError: Cannot access '_todayFmt' before initialization`.

The app cannot load at all in its current form. The fix is structural: either move the `Intl.DateTimeFormat` declarations above line 60, or move the query definitions after the formatters.

## Query staleness and silent cap

```javascript
const billsQuery = query(
  billsCollection,
  where("createdAt", ">=", getStartOfTodayTimestamp()),
  orderBy("createdAt", "asc"),
  limit(200)
);
```

This evaluates once at module load. The Firestore `onSnapshot` listener uses this fixed timestamp for the lifetime of the page. The `limit(200)` means a very busy day could silently drop bills from the snapshot — there's no UI indication that the cap was reached, and the user would see an incomplete bill list without knowing it.

## Catalog cache quota risk

The localStorage cache stores the full catalog as `{date: "YYYY-MM-DD", data: {...}}`. The `loadProducts` function has a top-level try/catch, but the `localStorage.setItem` call isn't individually wrapped. A `QuotaExceededError` would propagate to the outer catch, which shows a toast error and leaves `products` empty — meaning the app loads with no product catalog even though the Firestore fetch succeeded. Wrapping the `setItem` in its own try/catch (log and continue) preserves the fetch result.

## Stale draft cleanup race condition

```javascript
const now = Date.now();
if (now - _lastStaleCleanup > 30000) {
  _lastStaleCleanup = now;
  Object.keys(liveDraftsCache).forEach(id => {
    if (id !== sessionId && isDraftStale(liveDraftsCache[id])) {
      deleteDoc(doc(db, "liveDraftBills", id)).catch(() => {});
    }
  });
}
```

`isDraftStale` returns true if `updatedAt` is more than 2 minutes old. A session experiencing a brief network outage (phone locks screen, weak connectivity) will stop updating its draft's `updatedAt`. Another active tab will then delete it after 2 minutes, and when the original session reconnects, its live draft visibility to peers is gone without notification. The deletion is also multi-tab idempotent but generates unnecessary Firestore writes when multiple tabs race to delete the same stale draft.

## Debounced draft sync — 1-second deletion delay

`debouncedSyncLiveDraft()` wraps all draft syncs in a 1000ms debounce. When a bill is cleared (billItems becomes empty), the deletion of the live draft document is now delayed by 1 second. During that window, peers still see the draft in their live view. For a collaborative multi-device billing setup, this is a minor UX glitch where a "ghost" draft lingers briefly after the bill it represents has been submitted.
